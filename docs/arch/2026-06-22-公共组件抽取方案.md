# 公共组件抽取 —— 架构重构方案

**日期**：2026-06-22
**方案概述**：将 TCP-Chat UI 层中散落在各文件的重复图标、标签配置、UI 模式抽取为独立公共模块，消除代码重复，提升可维护性。同时按职责拆分 app.py（1700+ 行）为 Mixin 组合类。

---

## 1. 项目理解摘要

TCP-Chat 是一个基于 `tkinter` + `customtkinter` 的多房间聊天客户端。UI 层当前结构：

```
tcp_chat/ui/
├── theme.py          ✅ 已抽取：颜色/字体/尺寸常量
├── widgets.py        ✅ 已抽取：win_btn, make_draggable（仅 2 个组件）
├── app.py            ❌ 1700+ 行，包含大量内联重复
├── chat_page.py      ❌ 与 app.py 存在多处重复（tag 配置、滚动条逻辑）
├── start_page.py     ⚠️ 图标/样式硬编码
├── create_room_page.py ⚠️ 表单行构建模式内联
├── login_page.py     ⚠️ 表单行构建模式内联
├── initial_interface.py ⚠️ 与 app.py 重复窗口图标设置逻辑
├── file_transfer_ui.py  ⚠️ 消息插入模式与 app.py 重复
└── cache_manager.py  ✅ 独立模块，无需重构
```

---

## 2. 需求确认

### 2.1 需求复述

将以下公共要素从业务代码中抽取为独立模块，业务代码通过 import 调用：

| 公共要素 | 当前状态 | 影响范围 |
|----------|----------|----------|
| **图标/Emoji 常量** | 散落在 6+ 个文件中硬编码 | 全局 |
| **Text Tag 配置** | 在 `chat_page.py` 和 `app.py` 中各定义一次 | chat_page, app |
| **滚动条自动隐藏模式** | 在 `chat_page.py` 和 `app.py` 中重复实现 3 次 | chat_page, app |
| **消息插入模式** | `app.py` 3 个方法 + `file_transfer_ui.py` 2 个方法 | app, ft_ui |
| **窗口图标设置** | `app.py` 和 `initial_interface.py` 各实现一次 | app, initial |
| **确认弹窗模式** | `_confirm_disconnect` + `_show_ip` 中内联 | app |
| **表单行创建** | `login_page.py` 和 `create_room_page.py` 各实现一次 | login, create |

### 2.2 已确认决策

- **重构深度**：中度抽取（icons + tags + dialogs + patterns，约 6 个新文件，覆盖大部分重复模式）
- **app.py 拆分**：按职责拆分为多个 Mixin 类

#### 3.3.2 `tcp_chat/ui/tags.py` —— 文本标签配置

**职责**：集中定义 Text widget 的 tag 样式，提供统一的应用和缩放函数。消除 `chat_page.py:36-44` 和 `app.py:1256-1264` 的重复定义，以及 `app.py:488-498` 和 `app.py:1551-1558` 的重复缩放逻辑。

```python
"""
Text 标签配置 —— 消息区的 tag 样式定义与应用
"""
import tkinter as tk
from .theme import SYSTEM_FG, PRIVATE_FG, ERROR_FG, NICKNAME_FG, TIMESTAMP_FG, MSG_FONT_SIZE

# ── 标签配置表（唯一数据源） ──
TAG_DEFS: list[tuple[str, str, tuple]] = [
    ("system",       SYSTEM_FG,    ("Segoe UI", 9, "italic")),
    ("private",      PRIVATE_FG,   ("Segoe UI", 10)),
    ("error",        ERROR_FG,     ("Segoe UI", 10, "bold")),
    ("timestamp",    TIMESTAMP_FG, ("Segoe UI", 8)),
    ("nickname_tag", NICKNAME_FG,  ("Segoe UI", MSG_FONT_SIZE, "bold")),
    ("normal",       "#000000",    ("Segoe UI", MSG_FONT_SIZE)),
]


def apply_tags(text_widget: tk.Text, font_size: int = MSG_FONT_SIZE) -> None:
    """给 Text widget 一次性应用所有标准 tag 配置"""
    for name, fg, (family, size, *style) in TAG_DEFS:
        actual_size = font_size if name in ("normal", "nickname_tag") else max(8, int(font_size * 0.83))
        font_args = [family, actual_size]
        font_args.extend(style)
        text_widget.tag_configure(name, foreground=fg, font=tuple(font_args))


def scale_tags(text_widget: tk.Text, scale: float, base_size: int = MSG_FONT_SIZE) -> None:
    """按缩放比例重新配置所有 tag 字体"""
    msg_size = max(10, int(base_size * scale))
    overrides: dict[str, tuple] = {
        "normal":       ("Segoe UI", msg_size),
        "nickname_tag": ("Segoe UI", msg_size, "bold"),
        "system":       ("Segoe UI", max(8, int(10 * scale)), "italic"),
        "timestamp":    ("Segoe UI", max(7, int(9 * scale))),
    }
    for tag_name, font_val in overrides.items():
        text_widget.tag_configure(tag_name, font=font_val)
```

### 3.4 app.py 拆分方案 —— Mixin 模式

将 `ChatClientUI`（当前 1700+ 行）拆分为 1 个核心类 + 5 个 Mixin：

```
tcp_chat/ui/
├── app.py                  # ChatClientUI 核心（~300 行）
├── _tab_manager.py         # TabManagerMixin
├── _message_handler.py     # MessageHandlerMixin
├── _user_list_manager.py   # UserListManagerMixin
├── _command_menu.py        # CommandMenuMixin
└── _window_manager.py      # WindowManagerMixin
```

> 文件名以 `_` 前缀表示内部实现模块，不对外暴露。

#### Mixin 职责分配：

| Mixin | 行数(估) | 职责 | 关键方法 |
|-------|---------|------|---------|
| `TabManagerMixin` | ~250 | 标签页创建/切换/关闭/刷新 | `_switch_tab`, `_refresh_tabs`, `_create_private_tab_silent`, `_close_private_tab`, `_on_tab_right_click`, `_get_or_create_private_tab`, `_find_private_tab` |
| `MessageHandlerMixin` | ~300 | 消息解析/显示/路由/记录 | `_display_message`, `_add_system_message`, `_add_chat_message`, `_add_private_message`, `_route_private_message`, `_route_to_room_tab`, `_record_message`, `_rebuild_display_from_messages`, `_rebuild_private_chat_display` |
| `UserListManagerMixin` | ~200 | 在线用户解析/显示/右键菜单 | `_parse_user_list`, `_update_user_list_display`, `_user_cmenu`, `_toggle_block`, `_update_users_from_join_leave`, `_start_private_chat`, `_add_title_context_menu`, `_show_context_menu` |
| `CommandMenuMixin` | ~150 | 命令补全弹窗 | `_build_command_menu`, `_cmd_filter`, `_cmd_show`, `_cmd_hide`, `_on_cmd_key`, `_on_cmd_up/_down`, `_on_cmd_click`, `_cmd_insert_selected`, `_on_return` |
| `WindowManagerMixin` | ~150 | 窗口缩放/图标/拖拽/关闭 | `_set_window_icon`, `_on_window_resize`, `_apply_font_scaling`, `_on_close`, `_disconnect`, `_confirm_disconnect`, `_show_ip` |
| `ChatClientUI` (核心) | ~300 | 初始化/连接/生命周期协调 | `__init__`, `_build_user_interface`, `_switch_to_chat`, `_send_message`, `_process_queue`, `_connect_thread`, `_auto_connect`, `run`, 公开回调方法 |

#### 类定义示意：

```python
# app.py
from ._tab_manager import TabManagerMixin
from ._message_handler import MessageHandlerMixin
from ._user_list_manager import UserListManagerMixin
from ._command_menu import CommandMenuMixin
from ._window_manager import WindowManagerMixin

class ChatClientUI(
    TabManagerMixin,
    MessageHandlerMixin,
    UserListManagerMixin,
    CommandMenuMixin,
    WindowManagerMixin,
):
    """聊天室 GUI 主控 —— 组合所有 Mixin 行为"""

    def __init__(self, auto_connect=None):
        # Step 1: 初始化各 Mixin 的状态属性
        self._init_window_state()
        self._init_tab_state()
        self._init_message_state()
        self._init_user_list_state()
        self._init_command_menu_state()
        # Step 2: 核心初始化
        self.root = ctk.CTk()
        ...
```

**MRO (Method Resolution Order) 安全性**：
- 所有 Mixin 方法使用 `_` 前缀命名，确保不会意外覆盖
- 各 Mixin 的初始化通过独立命名的 `_init_*_state()` 方法完成，不依赖 `super().__init__()` 链
- 属性按 Mixin 分配前缀（如 tab 相关统一 `_tabs`/`_active_tab`，消息相关统一 `_msg_*`）

## 5. 风险评估与备选方案

### 方案 A：中度抽取 + Mixin 拆分（本次推荐方案）

**风险等级：中**

| 风险点 | 概率 | 影响 | 缓解措施 |
|--------|------|------|---------|
| 图标常量重命名导致遗漏 | 中 | 低 | 全局 Grep 验证每个图标引用点 |
| Mixin MRO 冲突 | 低 | 高 | Mixin 间不共享属性初始化，用独立 `_init_*_state()` 方法；方法名全部 `_` 前缀防覆盖 |
| 消息插入函数行为不一致 | 中 | 中 | 保留原方法作为 thin wrapper，逐步验证后移除 |
| 字体缩放 tag 重配逻辑变更 | 低 | 中 | `scale_tags` 提取自现有逻辑，行为等价；回归测试覆盖 |
| `_tab_manager.py` 跨 Mixin 引用 | 中 | 中 | 通过 `self` 访问，不做跨 Mixin 直接 import |

### 方案 B：仅抽取公共模块，不拆分 app.py（保守方案）

**风险等级：低**

仅新增 icons / tags / dialogs / patterns 四个公共模块，增强 widgets.py，不碰 app.py 内部结构。改动范围缩小 50%，只触及 6 个文件。

**优点**：风险极低，app.py 外部行为零变化，可独立上线
**缺点**：app.py 依然是 1700 行单文件，未解决可维护性根本问题

### 方案 C：全部抽取 + app.py 拆分 + 引入显式状态机

**风险等级：高**

在方案 A 基础上，引入显式状态机管理连接生命周期：

```
DISCONNECTED → CONNECTING → CONNECTED → IN_ROOM
                                      ↕
                                 DISCONNECTED
```

每个状态对应不同的 UI 行为（输入框启用/禁用、按钮状态等）。

**优点**：连接逻辑清晰，边界明确，适合后续扩展（如断线重连）
**缺点**：过度设计——当前状态转换简单且基本是单向的，状态机带来的复杂度大于收益。建议等连接逻辑变得更复杂时再引入。
