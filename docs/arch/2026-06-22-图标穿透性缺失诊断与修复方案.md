# 图标穿透性缺失诊断与修复方案

> 日期：2026-06-22 | 版本：v2（在统一图标管理器之后） | 风险：中

## 1. 方案概述

在 `icon_manager.py` 已实现（统一了 `set_window_icon` 接口、PNG→ICO 自动转换、CTkToplevel 兼容）的前提下，**窗口图标仍不显示自定义图标**，说明问题不在"是否调用了图标 API"，而在"调用的 API 不够完整"——特别是**缺少 `iconphoto()` 调用**导致任务栏图标失效，以及 **CTkToplevel 的 customtkinter 内部覆写**未被完全抑制。

```python
def set_window_icon(window):
```

### 2.3 Windows 图标机制对照

Windows 上 tkinter 图标的实际作用层级：

| API | 发送的 Win32 消息 | 影响范围 |
|-----|------------------|---------|
| `iconbitmap("x.ico")` | `SetClassLong(GCL_HICON)` + `WM_SETICON(ICON_BIG/SMALL)` | 标题栏小图标 + 任务栏大图标 |
| `wm_iconbitmap("x.ico")` | 同 `iconbitmap`，但让 CTkToplevel 置位 `_iconbitmap_method_called` | 同上 |
| `iconphoto(True, photo)` | `WM_SETICON(ICON_BIG/SMALL)`（从 PNG 生成 HICON） | 标题栏 + 任务栏 |
| .exe 内嵌图标 | 进程默认图标（由 `CreateProcess` 注册到 shell） | 任务栏分组图标 |
| `SetCurrentProcessExplicitAppUserModelID` | Windows 7+ 任务栏标识 | 任务栏分组图标 |

**关键发现：**

1. **`iconbitmap` 设置的是窗口类图标（class icon）**，Windows 会将其用于标题栏，但**不一定用于任务栏**
2. **`iconphoto` 设置的是窗口实例图标（window icon）**，在 Windows 上通过 `WM_SETICON` 直接发送 HICON，**对任务栏更可靠**
3. **最终谁有效取决于调用顺序** —— 最后调用的 `WM_SETICON` 覆盖前面的（因为操作的是同一个窗口属性）
4. **自定义 Toplevel 200ms 覆写机制**——CTkToplevel 创建后 200ms 会用默认图标覆写，除非 `_iconbitmap_method_called` 为 True

---

### 2.4 根因分析

#### 根因 1：`iconphoto()` 缺失导致的任务栏图标无效

当前 `set_window_icon` 只调 `iconbitmap()`/`wm_iconbitmap()`，不调 `iconphoto()`。

- **场景 A（源码运行 `python main.py`）**：进程是 `python.exe`，Windows 任务栏默认显示 Python 图标。`iconbitmap()` 改了窗口类图标，但任务栏分组图标由 shell 缓存（`python.exe` 的图标），不会更新。
- **场景 B（PyInstaller 打包的 exe）**：`.exe` 内嵌了 ICO，任务栏分组图标正确，但**CTkToplevel 子窗口**（初始界面、弹窗）的任务栏分组仍然可能显示默认图标。

#### 根因 2：customtkinter 对 CTkToplevel 的 200ms 图标覆写

customtkinter 的 `CTkToplevel.__init__` 在 `after_idle` 中注册了一个回调（`_after_idle_icon`），约 200ms 后用默认 tkinter 图标覆写窗口图标。它只在 `_iconbitmap_method_called` 为 True 时跳过覆写。

当前使用 `wm_iconbitmap()` 可以置位 `_iconbitmap_method_called`，但：
- 这仅在 `wm_iconbitmap()` 调用*早于* customtkinter 的 `after_idle` 触发时有效
- 如果初始化顺序导致 `set_window_icon()` 在 `after_idle` 之后才调用，覆写已经发生，后续的 `wm_iconbitmap()` 需要**再次置位**该标志才能生效

#### 根因 3：主窗口 withdraw → deiconify 过程中图标丢失窗口类注册

流程：
```
ChatClientUI.__init__()
  ├─ root = ctk.CTk()          → 窗口创建，类图标注册
  ├─ root.withdraw()           → 窗口隐藏
  ├─ set_window_icon(root)     → SetClassLong 修改了类图标
  └─ after(200, ...)
       └─ InitialInterface.show()
            └─ CTkToplevel(root) → 新窗口，不继承 root 的类图标
```

`withdraw()` 后 `deiconify()` 时，Windows 可能会重新读取窗口类图标，但 `SetClassLong` 修改的是已注册的类，不会丢失。但**关键是 `InitialInterface` 是一个独立 Toplevel**，它不继承 root 的类图标，需要单独设置。

#### 根因 4：`ensure_ico()` 的路径在某些情况下可能失效

`get_app_root()` 在 PyInstaller 打包后返回 exe 所在目录。如果 `.ico` 和 `.png` 不在 exe 所在目录（而是在 `_internal` 子目录中，对 `--onedir` 模式），`ensure_ico()` 会找不到文件。

## 4. 修复方案

### 4.1 核心修复：`set_window_icon` 增加 `iconphoto()` 调用

在 `icon_manager.py` 的 `set_window_icon()` 中，**在 `iconbitmap()` / `wm_iconbitmap()` 之前**先调用 `iconphoto()`：

```python
def set_window_icon(window) -> None:
    ico_path = ensure_ico()
    if not ico_path:
        return

    # ── Step 1: 加载 PNG 为 PhotoImage，调用 iconphoto ──
    # 这步注册大尺寸 HICON，Windows 任务栏优先使用此图标
    try:
        png_path = get_icon_path(".png")
        if os.path.exists(png_path):
            photo = tk.PhotoImage(file=png_path)
            window.iconphoto(True, photo)
    except Exception:
        pass

    # ── Step 2: 用 .ico 的 iconbitmap 覆盖 ──
    # iconbitmap 在后面调用，它的 SetClassLong 优先级高于 iconphoto 的 WM_SETICON
    try:
        if isinstance(window, ctk.CTkToplevel):
            window.wm_iconbitmap(ico_path)
            # 确保 _iconbitmap_method_called 被置位
        else:
            window.iconbitmap(ico_path)
    except Exception:
        pass
```

**为什么先 `iconphoto` 后 `iconbitmap`：**

| 调用顺序 | 标题栏 | 任务栏 | 原因 |
|---------|--------|--------|------|
| `iconbitmap` → `iconphoto` | ✅ 正常 | ✅ 正常 | `iconphoto` 的 `WM_SETICON` 覆盖 `iconbitmap`，但 `SetClassLong` 保留标题栏图标 |
| `iconphoto` → `iconbitmap` | ✅ 正常 | ✅ 正常 | **推荐**——`iconbitmap` 的 `SetClassLong` + `WM_SETICON` 完全覆盖，且 ICO 多尺寸兼容性更好 |
| 只有 `iconbitmap` | ✅ 正常 | ❌ 任务栏可能不生效 | **当前状态**——缺少 `iconphoto` 注册 HICON |
| 只有 `iconphoto` | ❌ 小尺寸模糊 | ✅ 正常 | `iconphoto` 从 PNG 生成 HICON，对小尺寸缩放可能不理想 |

### 4.2 加固：Windows 任务栏 AppUserModelID

在 `icon_manager.py` 增加模块级初始化，设置当前进程的 AppUserModelID，确保即使源码运行时任务栏也显示正确图标：

```python
def _set_app_user_model_id(app_id: str = "TCPChat.Room.1") -> None:
    """Windows 7+：设置当前进程的 Application User Model ID。
    
    这告诉 Windows 任务栏将当前进程归类到指定模型 ID，
    从而使用 .exe 或 .lnk 的图标，而不是 python.exe 的默认图标。
    
    只需在模块加载时调用一次。
    """
    if os.name != "nt":
        return
    try:
        import ctypes
        # SetCurrentProcessExplicitAppUserModelID
        # 原型: HRESULT SetCurrentProcessExplicitAppUserModelID(PCWSTR AppID)
        shell32 = ctypes.windll.shell32
        shell32.SetCurrentProcessExplicitAppUserModelID(app_id)
    except Exception:
        pass


# 模块加载时执行一次
_set_app_user_model_id()
```

### 4.3 加固：`set_window_icon` 再调用 + 延迟重试

对 CTkToplevel，customtkinter 的 200ms 覆写是**滞后触发器**，即使在创建时调用了 `wm_iconbitmap` 也不一定完全可靠。增加一个 `after(300)` 重设：

```python
# 在 InitialInterface.show() 中
set_window_icon(win)
# 额外保险：300ms 后重设一次，确保绕过 CTkToplevel 的 after_idle 覆写
win.after(300, lambda: set_window_icon(win))
```

或者在 `set_window_icon` 内部增加统一的延迟重试策略（更干净）。

### 4.4 修复：`ensure_ico()` 的 PyInstaller 打包路径兼容

当前 `get_app_root()` 在 PyInstaller `--onedir` 模式下返回 exe 所在目录。但打包后的文件布局可能是：

```
dist/TCP-Chat/
  ├── TCP-Chat.exe
  ├── _internal/        ← PyInstaller 的数据文件在此
  │   ├── TCP-Chat.png
  │   └── TCP-Chat.ico
  ├── config.json
  └── ...
```

如果图标文件在 `_internal/` 而代码在 `exe` 目录找，就找不到。

**修复：** `ensure_ico()` 增加对 `_internal/` 目录的回退搜索。

```python
def _find_icon_file(ext: str) -> Optional[str]:
    """搜索图标文件：先应用根目录，再回退到 _internal 目录（PyInstaller --onedir）。"""
    # 1. 应用根目录
    path = os.path.join(get_app_root(), f"TCP-Chat{ext}")
    if os.path.exists(path):
        return path
    # 2. _internal 子目录（PyInstaller --onedir 打包数据文件）
    internal = os.path.join(get_app_root(), "_internal", f"TCP-Chat{ext}")
    if os.path.exists(internal):
        return internal
    return None
```

## 8. 风险评估与备选方案

### 方案 A（推荐）：`iconphoto` + `iconbitmap` 双保险

- **风险**: 低
- **改动量**: 1 个文件（`icon_manager.py`）+ 1 行（`initial_interface.py`）
- **原理**: `iconphoto` 注册 `WM_SETICON`（大尺寸 HICON）→ `iconbitmap` 注册 `SetClassLong`（多尺寸 ICO 覆盖），两者互补
- **优点**: 同时解决标题栏和任务栏图标；兼容 CTkToplevel 的覆写机制
- **缺点**: 调用两次 API，增加约 5ms 延迟

### 方案 B：仅 `iconphoto()` + 放弃 `iconbitmap()`

- **风险**: 中
- **描述**: 统一只用 `iconphoto()`，传入 `icon_manager.py` 中从 PNG 生成的 `PhotoImage`
- **优点**: 代码最简单
- **缺点**: `iconphoto()` 在 Windows 上对标题栏 16×16 图标的缩放质量不稳定（PNG 缩放到 16×16 可能模糊）
- **适合**: 高 DPI 场景（任务栏优先）

### 方案 C：仅 `iconbitmap()` + 使用更高质量的 .ico

- **风险**: 中
- **描述**: 保持当前单 API 方案，但改进 .ico 生成质量（增加更多尺寸段、使用 32bpp RGBA）
- **优点**: 改动最小
- **缺点**: 仍不解决源码运行时任务栏图标为 python.exe 的问题；CTkToplevel 覆写风险仍在
- **适合**: 仅打包后使用（不关心源码运行场景）

### 方案 D：纯 Windows ctypes API + 完全跳过 tkinter 图标

- **风险**: 高
- **描述**: 直接用 `ctypes.windll.user32.SendMessageW` + `LoadImageW` 设置窗口图标
- **优点**: 完全控制，不受 tkinter/customtkinter 抽象层干扰
- **缺点**: Windows only；代码侵入性强；维护成本高；不跨平台
- **结论**: 方案 A 不够用的情况下才考虑此方案
