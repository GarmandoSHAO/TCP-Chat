## 1. 方案概述

将当前分散在 `app.py` 和 `initial_interface.py` 中的窗口图标设置逻辑，统一抽取到独立的 `tcp_chat/ui/icon_manager.py` 模块中。所有窗口（主窗口、初始界面、弹窗）统一调用该模块。以 `TCP-Chat.png` 为图标源文件，利用 Pillow 自动同步生成 `.ico`，修改图标只需替换一张 PNG。

### 4.2 模块职责

| 职责 | 说明 |
|------|------|
| PNG → ICO 自动转换 | 检测 `TCP-Chat.png` 修改时间，自动同步生成多尺寸 `TCP-Chat.ico` |
| 统一 API | `set_window_icon(window)` — 自动判断窗口类型选正确 API |
| CTkToplevel 兼容 | 自动使用 `wm_iconbitmap` 绕过 customtkinter 的 200ms 图标覆写 bug |
| 防御性降级 | Pillow 不可用时降级使用已有 `.ico`；两者皆无时静默跳过 |
| 日志记录 | 每次调用打一条 INFO 级别日志，方便排查图标问题 |

### 4.3 接口设计

```python
import os
import tkinter as tk
from typing import Optional, Union
import customtkinter as ctk

# ==================== 公开接口 ====================

def set_window_icon(window: Union[ctk.CTk, ctk.CTkToplevel, tk.Tk, tk.Toplevel]) -> None:
    """对任意窗口设置 TCP-Chat 图标。

    内部自动判断窗口类型：
      - Windows + CTkToplevel → win.wm_iconbitmap(ico_path)
      - CTk / tk.Tk           → win.iconbitmap(ico_path)
      - tk.Toplevel           → win.iconphoto(True, photo_img)
      - 非 Windows            → win.iconphoto(True, photo_img)

    Args:
        window: 需要设置图标的窗口实例。
    """

def get_icon_path(ext: str = ".ico") -> str:
    """获取图标文件的绝对路径。

    Args:
        ext: 文件扩展名，如 ".ico" 或 ".png"

    Returns:
        图标文件的绝对路径字符串。
    """

def ensure_ico() -> Optional[str]:
    """确保 .ico 文件存在且为最新，返回其路径。

    如果 TCP-Chat.png 比 TCP-Chat.ico 新（或 .ico 不存在），
    且 Pillow 可用，则自动重建 .ico。
    如果 Pillow 不可用但 .ico 已存在，直接返回 .ico 路径。
    如果 .ico 和 .png 都不存在，返回 None。

    Returns:
        .ico 文件的绝对路径，或 None。
    """

# ==================== 内部函数 ====================

def _png_to_ico(png_path: str, ico_path: str) -> bool:
    """用 Pillow 将 PNG 转为多尺寸 ICO 文件。

    生成的 ICO 包含以下尺寸：16×16, 32×32, 48×48, 64×64, 128×128, 256×256。
    转换失败时静默返回 False，不抛异常。

    Args:
        png_path: 源 PNG 文件路径。
        ico_path: 目标 ICO 文件路径。

    Returns:
        转换成功返回 True，否则 False。
    """

def _need_rebuild(png_path: str, ico_path: str) -> bool:
    """检查 .ico 是否需要重建。

    需要重建的条件：
      - .ico 文件不存在
      - .png 文件的修改时间晚于 .ico 文件的修改时间

    Args:
        png_path: PNG 文件路径。
        ico_path: ICO 文件路径。

    Returns:
        需要重建返回 True。
    """

# ==================== 模块常量 ====================

_ICO_SIZES = [16, 32, 48, 64, 128, 256]
"""生成 ICO 时包含的尺寸列表。"""

try:
    from PIL import Image
    _HAS_PIL = True
except ImportError:
    _HAS_PIL = False
"""Pillow 可用性标志。"""

_logger = None
def _get_logger():
    global _logger
    if _logger is None:
        import logging
        _logger = logging.getLogger(__name__)
    return _logger
```

### 4.4 模块内部流程图

```
set_window_icon(window)
  │
  ├─ 1. 调用 ensure_ico() 获取 ICO 路径
  │     │
  │     ├─ ico_path = os.path.join(get_app_root(), "TCP-Chat.ico")
  │     ├─ png_path = os.path.join(get_app_root(), "TCP-Chat.png")
  │     │
  │     ├─ _need_rebuild(png_path, ico_path)?
  │     │     ├─ 是且 _HAS_PIL → _png_to_ico(png_path, ico_path)
  │     │     └─ 否 → 跳过
  │     │
  │     └─ return ico_path (或 None)
  │
  ├─ 2. ico_path 为 None? → return (静默跳过)
  │
  ├─ 3. 判断窗口类型
  │     ├─ isinstance(window, ctk.CTkToplevel)
  │     │     └─ window.wm_iconbitmap(ico_path)
  │     │        # wm_iconbitmap 绕过 CTkToplevel._iconbitmap_method_called
  │     │
  │     ├─ isinstance(window, (ctk.CTk, tk.Tk))
  │     │     └─ window.iconbitmap(ico_path)
  │     │
  │     └─ else (tk.Toplevel 等)
  │           └─ window.iconphoto(True, _load_photo(png_path))
  │
  └─ 4. 打日志: f"set_window_icon({type(window).__name__}) → OK/FAILED"
```

## 6. 关键问题决策

### 6.1 为什么用 Pillow 而不用其他方法？

| 方法 | 结论 |
|------|------|
| `tk.PhotoImage(file="xxx.png")` + `iconphoto()` | CTkToplevel 的 `_iconbitmap_method_called` 不识别 iconphoto，200ms 后图标被覆盖 |
| `windll.shell32.SetCurrentProcessExplicitAppUserModelID` | 只影响任务栏，不影响窗口标题栏 |
| 直接引用 .ico 不改代码 | 用户需求明确指向 .png 为源 |
| **Pillow 自动转 ICO** | **兼容所有情况，且只改一个 PNG 就生效** |

### 6.2 CTkToplevel 的图标覆写问题

customtkinter 在 CTkToplevel 创建约 200ms 后（`_after_idle_icon`）会用默认图标重设窗口图标，除非内部标志 `_iconbitmap_method_called` 被置位。该标志仅在 `wm_iconbitmap()` 被调用时才会置位。因此对 CTkToplevel 必须使用 `wm_iconbitmap()` 而非 `iconbitmap()`。

## 7. 风险评估

| 风险 | 等级 | 概率 | 缓解措施 |
|------|------|------|---------|
| Pillow 安装失败 | 低 | 低 | 防御性导入，降级使用已有 .ico |
| Pillow 安装体积大 | 中 | 中 | 已是 Python GUI 项目标配；约 10MB |
| PNG 转 ICO 尺寸不对劲 | 低 | 低 | 生成 6 个标准尺寸覆盖所有场景 |
| 图标路径在 PyInstaller 打包后失效 | 低 | 低 | 沿用 `get_app_root()`，已处理 frozen 场景 |
| 用户只改 PNG 不重启 | 低 | 低 | 启动时自动检测 mtime 重建 ICO |
| 删除 .ico 但无 Pillow | 低 | 低 | 静默跳过，不影响程序运行 |

## 8. 备选方案

### 方案 B：纯 tk.PhotoImage + iconphoto（无 Pillow）

| 维度 | 评估 |
|------|------|
| 方法 | `tk.PhotoImage(file="TCP-Chat.png")` + `window.iconphoto(True, img)` |
| 优点 | 不新增依赖 |
| 缺点 | ① CTkToplevel 图标覆写 bug 仍需要 `wm_iconbitmap` 绕过，但 `wm_iconbitmap` 不接受 PNG ② 仅 Windows 上 `iconbitmap` 不接受 PNG ③ 多分辨率支持差 |
| **风险等级** | **中** — CTkToplevel 的图标会被覆盖，主窗口也只能在非 Windows 正常工作 |

### 方案 C：纯 ICO 方案，仅统一代码不改 PNG 源

| 维度 | 评估 |
|------|------|
| 方法 | 不引入 Pillow，统一引用 `TCP-Chat.ico`，只重构代码不改变图标源 |
| 优点 | 零依赖，改动最小 |
| 缺点 | 不符合"以 PNG 为源"的需求，改图标仍需手动准备 `.ico` |
| **风险等级** | **低** — 但如果用户想只换 PNG，此方案不满足需求 |

### 方案 D：外部脚本一次性生成 ICO，运行时不做转换

| 维度 | 评估 |
|------|------|
| 方法 | 提供一个 `tools/generate_icon.py` 脚本，用户修改 PNG 后手动运行一次 |
| 优点 | 运行时无 Pillow 依赖 |
| 缺点 | 多一步手动操作，"方便修改"打折扣 |
| **风险等级** | **低** — 但用户体验不如自动转换 |
